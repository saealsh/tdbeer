## 🎯 ماذا تم إصلاحه؟

### قبل الإصلاح
لو شبكة المستخدم ضعيفة أو انقطعت لثانيتين عند إرسال رسالة:
- ❌ الرسالة **تضيع تماماً**
- ❌ المستخدم يشوف "فشل الإرسال" ويحتاج يكتبها من جديد
- ❌ نفس الشي يصير لطلبات الصداقة

### بعد الإصلاح
- ✅ **3 محاولات تلقائية** مع backoff (500ms, 1s, 2s)
- ✅ لو فشلت كلها، الرسالة تُحفظ في WriteQueue وتُرسل لما يرجع النت
- ✅ لو المستخدم أوفلاين أصلاً، الرسالة تُحفظ مباشرة بدون محاولة
- ✅ Network module يعرض Toast تلقائياً ("📴 ما فيه اتصال" / "🌐 رجع الاتصال")

## 📋 تفاصيل المنطق الجديد

### `sendMessage` — النمط التدريجي

```
هل المستخدم أونلاين؟
├── نعم → withRetry(3 محاولات)
│   ├── نجح → ✅ Toast "تم"، صوت إرسال، اهتزاز
│   └── فشل → fallback إلى WriteQueue + Toast "راح يُعاد المحاولة"
└── لا → احفظ مباشرة في WriteQueue + Toast "محفوظة، راح تُرسل لما يرجع النت"
```

**ملاحظة تقنية:** في حالة الأوفلاين، نستخدم `Date.now()` كـ timestamp بدلاً من `serverTimestamp()`. السبب: `serverTimestamp` كائن خاص ما يقدر JSON.stringify يحفظه. الفرق غالباً أقل من ثانية، ما يأثر على ترتيب الرسائل.

### `sendFriendRequest` و `acceptFriendRequest` — withRetry فقط

هذي عمليات يقلّ احتمال أن المستخدم يعملها أوفلاين، فاكتفينا بـ `withRetry` بدون WriteQueue. لو فشلت بعد 3 محاولات، يشوف Toast "فشل" ويعيد الضغط على الزر بنفسه.

## 🧪 كيف تختبر

### اختبار 1: الشبكة الضعيفة
1. افتح Chrome DevTools → Network → Throttling → "Slow 3G"
2. أرسل رسالة لصديق
3. لازم تشوف: الرسالة تنرسل بنجاح حتى لو أخذت 3-5 ثواني

### اختبار 2: الأوفلاين الكامل
1. DevTools → Network → "Offline"
2. أرسل رسالة
3. لازم تشوف: Toast "📴 الرسالة محفوظة — راح تُرسل لما يرجع النت"
4. شغّل النت تاني
5. لازم تشوف: Toast "☁️ تم مزامنة 3 تغيير"

### اختبار 3: تحقق من الطابور
في Console:
```javascript
window.WriteQueue.size()   // عدد العمليات في الطابور
window.WriteQueue.flush()  // أجبر المزامنة الآن
```

## ⚠️ قيود معروفة

1. **عدّاد unread في الأوفلاين**: لما المستخدم يرسل رسالة وهو أوفلاين، ما نقدر نقرأ unread الحالي للطرف الآخر. نضع 1 افتراضياً. لو الطرف الآخر عنده 5 رسائل غير مقروءة، راح يصير 1 بدل 6. **تأثير محدود — يصلح نفسه عند المزامنة التالية.**

2. **ترتيب الرسائل**: في حالة الأوفلاين الطويل، الرسائل المحفوظة قد تظهر بترتيب `Date.now()` المحلي بدلاً من ترتيب السيرفر. **ما تخسر رسائل، فقط الترتيب قد يختلف بثانية.**

## ✨ النتيجة

تطبيقك الحين عنده **resilience** حقيقي:
- المستخدمون في الأماكن النائية (شبكة ضعيفة) ما يخسرون رسائلهم
- التطبيق يعمل أوفلاين ويزامن لما النت يرجع
- تجربة مستخدم أفضل بكثير على شبكات الموبايل غير المستقرة

النشر آمن — الكود يعتمد على fallbacks في كل مكان، لو أي module ما تحمّل الكود يرجع للسلوك الأصلي.
